# Frontend Anti-Patterns

本文件列出 AI 生成前端时最常见、最容易导致“能跑但不好用”的反模式。以下内容不是建议，而是默认禁止项；若确有例外，必须在项目级 `docs/frontend-ui-checklist.md` 中说明理由。

## 1. 裸写原生业务控件

禁止：

- 在页面中直接裸写原生 `select`、`button`、`input` 作为最终业务控件
- 同一项目中混用多套风格不一致的原生表单控件
- 为了省事在单页里临时写一个“简化版按钮/下拉框”

正确做法：

- 优先复用统一 UI 组件，如 `Button`、`Input`、`Select`
- 若组件库缺失，先补通用组件，再在页面里使用

## 2. 按钮文本换行或按钮被压缩变形

禁止：

- 操作按钮文本自动断成两行
- 图标和文字被挤压到垂直不对齐
- 用容器挤压去“适配”窄宽度

正确做法：

- 按钮默认使用单行策略，例如 `whitespace-nowrap`
- 按钮容器应避免被压缩，例如 `shrink-0`
- 一行按钮过多时，改为换行布局、DropdownMenu 或分组操作

## 3. 长文本撑坏布局

禁止：

- 长 Skill 名称、分类名、作者名、版本号直接把卡片或操作区撑坏
- 排行项、列表项、审核项中长文本把按钮顶出可视区
- 对标题、Badge、辅助文案完全不做截断或换行策略

正确做法：

- 长文本容器显式设置 `min-w-0`
- 根据场景选择 `truncate`、受控换行或 tooltip
- 同排混合“文本 + 按钮/状态”时，先保证操作区稳定，再处理文本截断

## 4. 只写比例列，不写最小宽度

禁止：

- 筛选栏、操作栏、表单行只写 `1fr 1fr` 或百分比比例，不设最小宽度
- 在窄宽度下仍强行并排多个控件

正确做法：

- 使用 `minmax(...)` 或明确最小宽度
- 在断点下主动重排为两列或单列
- 先保证控件完整展示，再追求横向紧凑

## 5. 用默认浏览器样式交差

禁止：

- 白底灰边原生下拉框直接上线
- 不统一的圆角、边框、阴影和控件高度混在同一页面
- 视觉层级完全依赖浏览器默认样式

正确做法：

- 所有核心控件使用统一 token
- Select、Input、Textarea、Button 高度、边框和阴影统一
- 在项目级 `docs/frontend-ui-checklist.md` 里记录控件复用情况

## 6. 没有为极端内容做保护

禁止：

- 假设分类名永远很短
- 假设按钮文字永远是 2-3 个字
- 假设列表只会出现短标题、短描述

正确做法：

- 生成时默认按“真实业务数据会更长”处理
- 对长标题、长描述、长分类、长版本号、长按钮文案做约束
- 在自查中至少检查一次长内容场景

## 7. 只过 `build` / `lint`，不做视觉自查

禁止：

- 认为构建通过就代表页面没问题
- 不检查按钮换行、控件变形、长文本溢出、筛选栏错位

正确做法：

- 对照 `docs/frontend-ui-spec.md` 中的验收清单做一轮视觉与可用性自查
- 至少覆盖：
  - 按钮是否换行
  - 下拉框是否变形
  - 长文本是否撑坏布局
  - 窄屏下筛选栏和操作区是否还能用

## 8. 页面内联样式和一次性修补过多

禁止：

- 在页面中堆大量临时 class 只修当前页面
- 同类控件在不同页面各修一次

正确做法：

- 优先沉淀到通用组件
- 能在 `Button`、`Select`、`Card`、布局组件里解决的，不在页面层重复修

## 9. 受限点击静默失败

禁止：

- 未登录时点击收藏、安装、提交、进入工作台等操作完全没反应
- 无权限时只把按钮做成可点击外观，但点击后不提示原因
- 缺少前置条件时既不禁用也不提示，用户只能反复点击猜测

正确做法：

- 未登录时给出登录提示，并跳转或引导到登录
- 无权限时明确说明限制原因
- 缺少前置条件时说明缺什么，并提供下一步动作
- 如果控件禁用，仍要在附近提供原因说明，不要只靠灰态

## 10. 用假交互伪装业务闭环

禁止：

- 核心创建、提交、审核、发布、安装、评价等动作只执行 `setTimeout`、静态 toast 或本地状态变更
- 管理页、工作台、统计卡片、分类筛选使用硬编码业务数据，却在界面上表现为真实结果
- API 尚未实现时让按钮显示“成功”，但后端没有任何状态变化

正确做法：

- 核心动作必须调用统一 API client、TanStack Query mutation 或 typed domain hook
- 缺 API 时显示能力暂不可用，并在 OpenSpec/tasks 或项目级 checklist 标记为 Open
- 成功 toast 必须发生在真实 mutation 成功之后，失败 toast 必须来自真实错误处理路径

## 11. 认证态只有模块变量

禁止：

- 只把 access token 存在模块级变量中，刷新页面后登录态丢失且没有说明
- 有注册接口但没有注册入口，或有登录页但没有 AuthContext/会话状态/退出登录
- 401 只在控制台输出，页面没有重新登录、刷新 token 或权限提示

正确做法：

- 提供 AuthContext、auth store 或等价会话层，统一处理登录、登出、当前用户、401 与 refresh 策略
- 若不做 refresh token，必须使用短 access token，并在 `docs/security-notes.md` 说明风险与取舍
- 注册、登录、退出和受限动作提示必须能被测试或 smoke 验证覆盖

## 快速自查问题

提交前至少问自己：

- 页面里是否还存在裸写的原生 `select` / `button` / `input`
- 是否存在任何按钮换行
- 是否存在任何控件高度异常、箭头错位、文本挤压
- 是否存在长文本把卡片、筛选栏、操作区撑坏
- 是否只在桌面端看起来正常，窄屏就失稳
- 是否存在用户点击后无反应，且没有登录提示、权限提示或前置条件说明
- 是否存在核心按钮只用 `setTimeout`、toast 或本地假数据完成
- 是否存在硬编码统计、分类、审核结果或工作台数据冒充后端结果
- 是否存在认证态只有模块变量、没有会话层或 401 处理
