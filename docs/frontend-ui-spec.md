# Frontend UI Spec

本文件是模板仓库的前端 UI 总入口规范。目标不是定义某个业务项目的视觉风格，而是提供一套对多数后台、工作台、管理台、工具平台都通用且可执行的前端生成约束。

如果你只读一份前端规范，优先读本文件；如果需要更细的展开，再继续看以下配套文档：

- `docs/design-tokens.md`：默认 token 参考值，不定义行为规则
- `docs/component-patterns.md`
- `docs/frontend-anti-patterns.md`

## 1. 目标

前端生成必须同时满足三件事：

- 可用：关键业务动作真实可执行，而不是只有静态页面
- 稳定：长文本、窄宽度、加载态和异常态下不走样
- 可维护：通用控件和布局可复用，不靠页面级临时修补维持

本规范默认适用于以下常见类型：

- 后台管理台
- 内部工具平台
- 运营工作台
- 列表 + 详情 + 表单 + Dashboard 组合型 Web 应用

## 2. 通用实现原则

- 先保证业务闭环，再做视觉修饰
- 先定义统一控件和布局规则，再写页面
- 先考虑真实数据长度和状态变化，再写静态排版
- 先保证桌面端和移动端都成立，再局部微调

## 3. 视觉表达原则

- 界面应有明确的信息层级、操作层级和状态反馈
- 风格要稳定统一，不允许每个页面各自为政
- 优先输出可以继续扩展的设计系统，而不是一次性页面拼装
- 页面必须看起来像可交付产品，而不是默认控件堆叠
- 页面必须真实连接业务 API 或业务 action hook；核心按钮不能只弹 toast、`setTimeout` 或更新本地假数据

### 3.1 主题基础

生成前端时，至少先定义以下主题 token 或 CSS 变量：

- 颜色：主色、强调色、成功、警告、危险、中性色、边框色、背景层级
- 字体：标题、正文、辅助文本的字号与字重
- 间距：至少定义 4-6 个常用 spacing 档位
- 圆角：按钮、输入框、卡片、弹窗应有统一圆角规则
- 阴影：悬浮层、卡片、下拉菜单应有统一阴影层级
- 断点：移动端、平板、桌面端至少三档

默认 token 参考值见 `docs/design-tokens.md`，项目可按业务需求覆盖。

禁止直接把所有颜色和 spacing 写死在单个组件中。

### 3.2 页面层级

每个关键页面至少应体现：

- 页面标题与说明区
- 主操作区
- 内容主体区
- 辅助信息或筛选区
- 空态、错误态、加载态区域

不要把所有信息压成一块白底容器，也不要全站都是同一层级的卡片堆叠。

### 3.3 视觉表达重点

- 信息层级是否明确
- 页面节奏是否稳定
- 主题 token 是否被一致使用
- 卡片、标题、说明文字、辅助信息是否形成清晰主次
- 页面是否看起来像可交付产品，而不是默认控件堆叠

### 3.4 审美底线

- 不要只输出默认白底、灰边、普通蓝按钮的无识别界面
- 不要所有页面共用完全一样的头图、卡片和布局节奏
- 不要把“精美”理解为堆渐变、堆阴影或堆动画
- 允许适度品牌化，但不能牺牲信息清晰度和可用性

## 4. 必须统一的内容

生成前端时，以下内容必须统一，不得页面各自为政：

- 颜色、字号、间距、圆角、阴影、断点
- Button、Input、Textarea、Select、Card、Badge、Toast、Skeleton
- 加载态、空态、错误态、禁用态、提交中态、成功反馈
- 主操作、次操作、危险操作的视觉层级
- HTTP 请求封装、错误处理与未授权提示

其中 token 数值本身以 `docs/design-tokens.md` 为默认参考；是否需要这些 token、如何统一使用，则由本文件约束。

## 5. 必须覆盖的页面结构

常见页面必须至少符合以下结构：

- 列表页：标题区、筛选区、结果区、状态区
- 详情页：摘要区、主内容区、关键操作区
- 表单页：说明区、字段分组、字段级错误、底部操作区
- 工作台页：摘要指标、待处理事项、快捷操作、最近活动或重点区域

### 5.1 列表页

建议包含：

- 页头：标题、说明、主操作按钮
- 筛选区：搜索、筛选、排序、批量操作
- 结果区：表格或卡片列表
- 状态区：加载、空态、错误态

常见错误：

- 只有表格，没有筛选和主操作
- 空态只写 “No data”
- 状态标签、时间、负责人等关键信息没有层级

### 5.2 详情页

建议包含：

- 头部摘要：名称、状态、关键指标、关键操作
- 主内容区：基础信息、时间线、关联数据
- 辅助区：说明、审计信息、备注、快捷操作

常见错误：

- 详情页只是字段平铺
- 缺少状态变化入口
- 关键操作藏得过深

### 5.3 表单页

建议包含：

- 页面说明：当前动作和影响范围
- 表单主体：按业务分组
- 辅助说明：字段解释、约束说明
- 底部操作：取消、保存、提交等

常见错误：

- 所有字段一次性堆满
- 错误提示不贴近字段
- 提交中和成功反馈缺失

### 5.4 工作台 / Dashboard

建议包含：

- 顶部摘要指标
- 待处理事项区
- 最近活动或关键提醒
- 快捷操作入口

常见错误：

- 只有统计卡片，没有可操作入口
- 所有模块权重相同，缺乏主次

### 5.5 空态和错误态

空态至少应包含：

- 当前为空的原因
- 下一步建议动作
- 返回或创建入口

错误态至少应包含：

- 错误说明
- 重试动作
- 返回路径或联系支持提示

## 6. 通用控件硬规则

- 禁止在业务页面中直接裸写原生 `select`、`button`、`input` 作为最终控件
- 所有操作按钮默认不得换行
- 长标题、长分类名、长版本号、长排行名称必须显式处理截断或换行策略
- 同排混合“长文本 + 按钮/状态”时，文本容器必须显式支持收缩，例如 `min-w-0`
- 筛选栏、操作栏、表单行必须声明最小宽度、`minmax(...)` 或断点重排策略

具体反例见 `docs/frontend-anti-patterns.md`。

## 7. 状态设计硬规则

每个关键页面都必须显式处理：

- 首次加载
- 局部加载
- 空数据
- 请求失败
- 表单校验失败
- 提交中
- 提交成功
- 禁用或无权限
- 未登录
- 缺少前置条件，例如未选择对象、未完成绑定、未安装依赖资源

不得出现：

- 点击后无反馈
- 写操作成功或失败时静默
- 空页面只写 “No data”
- 用户点击受限操作后没有任何提示、解释或下一步引导
- 核心写操作只用 `setTimeout`、静态 toast、硬编码成功文案模拟成功
- 统计、分类、审核列表、我发布/我安装等业务数据硬编码为固定值

本节不仅要求状态存在，也要求状态对用户可见、可理解、可反馈。

对于未登录、无权限或缺少前置条件的点击行为，前端必须至少满足以下之一：

- 直接跳转到登录页，并提示当前操作需要登录
- 弹出登录弹窗、权限说明或前置条件说明
- 使用 Toast、Inline Message 或 Tooltip 明确说明为什么当前不能执行
- 在控件默认禁用时，同时提供可见说明，而不是只让按钮变灰

## 8. 响应式硬规则

- 多列布局在窄屏下必须重排
- 筛选区和操作区在窄屏下不得出现按钮断裂、下拉框变形、箭头错位
- 表格类复杂内容必须明确横向滚动、卡片化或折叠策略
- 移动端主要操作必须可见、可点、可理解
- 表单在移动端不能出现难以点击或被遮挡的问题

## 9. 组件与交互模式

默认应覆盖以下模式：

- ErrorBoundary
- 路由级 `React.lazy + Suspense`
- Skeleton
- Empty State
- Toast
- Confirm Dialog
- 按钮变体
- 表单提交中态
- 响应式导航或响应式操作区

细节见 `docs/component-patterns.md`。

### 9.1 API 对接完整性

前端生成必须把需求中的关键业务页面映射到真实 API：

- 列表页：筛选、分类、排序、分页参数必须进入真实 query；分类选项优先来自分类 API，不得固定写死业务分类。
- 详情页：安装、评价、发布版本、状态变更等写操作必须使用 mutation，并处理成功、失败、提交中、未登录和无权限状态。
- 工作台页：我发布的、我安装的、创建 Skill、提交审核、发布版本必须调用真实 API 或明确的 typed hook。
- 管理页：待审核列表、通过、拒绝、下架、运营统计必须调用真实 admin API；不得使用硬编码数字和 toast-only 动作。
- 认证页：需求包含注册时必须提供注册入口；登录成功后必须有 AuthContext 或等价全局认证状态，刷新/401/登出策略必须明确。

如果某个 API 尚未实现，前端不得伪造成可用功能；必须显示”暂不可用”状态并在 `docs/frontend-ui-checklist.md` 标为 Open。

### 9.2 列表页筛选与分页接线

列表页的搜索、分类筛选、排序和分页控件必须与 API 查询参数真实联动：

- 每个筛选控件必须绑定受控 state 或 URL search params
- state 变化必须驱动 `useQuery` 重新请求
- 后端返回分页元数据（total/page/page_size）时，前端必须渲染分页控件
- 查询参数不得硬编码为固定值

审计脚本会检测列表页（文件名含 Market/List/Browse 等）中的 `<input>` 和 `<select>` 是否有对应的 state 绑定。无状态绑定的控件视为装饰性控件，会触发审计失败。详见 `docs/frontend-anti-patterns.md` §12。

## 10. 明确禁止的反模式

以下不是“尽量避免”，而是默认禁止：

- 裸写原生业务控件
- 按钮文本换行或按钮被挤压变形
- 长文本撑坏布局
- 只写比例列，不写最小宽度
- 默认浏览器样式直接上线
- 只过 `build` / `lint`，不做视觉自查
- 页面层反复打补丁，不沉淀到通用组件

完整列表见 `docs/frontend-anti-patterns.md`。

## 11. 生成时的最小执行顺序

生成前端时，建议至少按这个顺序执行：

1. 读取本文件 `docs/frontend-ui-spec.md`
2. 读取 `docs/design-tokens.md`
3. 读取 `docs/component-patterns.md`
4. 读取 `docs/frontend-anti-patterns.md`
5. 结合本文件中的页面结构要求组织布局
6. 生成统一 UI 组件和页面布局
7. 再生成具体业务页面
8. 最后对照本文件中的验收清单自查

## 12. 验收标准

前端交付前至少应满足：

- `npm run build` 通过
- `npm run lint` 通过
- 页面分层合理，`App.tsx` 不承载全部业务逻辑
- 关键页面覆盖必要状态
- 没有明显按钮换行、控件变形、长文本溢出
- 桌面端和移动端布局都可用
- 至少存在一处 ErrorBoundary、关键路由懒加载和统一 HTTP 错误处理
- `npm test -- --run` 通过，且测试覆盖至少一个关键页面状态或表单校验
- 关键业务按钮调用真实 API/action hook，不存在 `setTimeout`、静态 toast-only、硬编码统计数字或硬编码业务分类
- Vite `index.html` 包含 doctype、lang、charset、viewport 和 title
- 前端依赖有 `package-lock.json`、`pnpm-lock.yaml` 或 `yarn.lock`

### 12.1 结构检查

- 页面是否按 `pages/`、`features/`、`components/`、`api/` 等分层
- `App.tsx` 是否只负责装配
- 是否存在明显的大型单文件组件

### 12.2 视觉一致性检查

- 是否定义了主题 token 或 CSS 变量
- 颜色、圆角、阴影、间距是否统一
- 主按钮、次按钮、危险按钮是否有清晰区分

### 12.3 页面质量检查

- 列表页是否有标题、筛选、结果区、状态区
- 详情页是否有摘要区和关键操作
- 表单页是否有字段分组、字段级错误提示和提交反馈
- 工作台页是否不只是 KPI 数字堆叠

### 12.4 状态完整性检查

- 是否补齐加载态、空态、错误态、禁用态、提交中态、成功反馈
- 接口失败时页面是否有明确提示
- 空数据时是否有下一步引导
- 未登录、无权限、缺少前置条件时，点击是否会给出明确提示或跳转引导
- 是否通过统一 HTTP 层处理接口错误，而不是在页面中零散吞错

### 12.5 响应式与可用性检查

- 移动端是否可操作
- 多列布局是否会在窄屏重排
- 表格或复杂列表在移动端是否有降级方案

### 12.6 反模式排查

- 是否踩中了 `docs/frontend-anti-patterns.md` 中列出的禁止项
- 是否仍存在裸写原生业务控件、按钮换行、长文本撑坏布局等明显问题

### 12.7 风险记录

生成项目时，建议在 `generated/<project-slug>/docs/frontend-ui-checklist.md` 中记录：

- 已满足项
- 未满足项
- 视觉风险
- 交互风险
- ErrorBoundary / 懒加载 / HTTP 错误处理检查结果
- 待人工确认项

## 13. 使用方式

在 prompt、skill 或人工协作中，可直接引用本文件作为总入口，例如：

```text
先读取 docs/frontend-ui-spec.md，再按其中引用的细分文档生成前端；禁止踩 docs/frontend-anti-patterns.md 中列出的反模式。
```
