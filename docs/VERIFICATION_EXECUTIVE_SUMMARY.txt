1. THREE-LAYER VERIFICATION ARCHITECTURE
================================================================================

The template implements a hierarchical verification system:

Layer 1: TEMPLATE-LEVEL AUDIT (audit_generated_project.sh)
  ├─ When: Immediately after generation
  ├─ Runtime: ~2-5 seconds (no execution)
  ├─ What: Structural validation, config keys, code patterns, forbidden content
  ├─ Exit: YES (fails fast if structure incomplete)
  └─ Purpose: Verify generation completed correctly

Layer 2: PROJECT-LEVEL VERIFICATION (verify_project.sh)
  ├─ When: After template audit passes
  ├─ Runtime: ~2-5 min (default) or ~5-10 min (--with-compose-up)
  ├─ What: Test execution, lint/build success, quality gates
  ├─ Exit: YES (fails on test/lint/build failure)
  └─ Purpose: Verify project quality and executability

Layer 3: PROJECT-INTERNAL SCRIPTS (verify_all.sh, check_business_flow.sh)
  ├─ When: During development, after services running
  ├─ Runtime: Varies
  ├─ What: One-shot verification, OpenAPI export, business flow testing
  ├─ Exit: YES (for error conditions)
  └─ Purpose: Project-specific automation and validation

  ├─ Does NOT run business flow
  ├─ Typical duration: 2-5 min

  ├─ Prerequisite: Step 2 passed
  ├─ Starts services (docker compose up)
  ├─ Runs business flow (if check_business_flow.sh exists)
  ├─ Cleanup: Auto docker compose down
  ├─ Typical duration: 5-10 min
  └─ Exit: YES

Step 4 (Optional): Project's internal scripts
  ├─ Prerequisite: Step 3 (or 3a/3b) passed
  ├─ Can run anytime: verify_all.sh, export_openapi.sh
  ├─ Requires services: check_business_flow.sh
  └─ Exit: YES

DEPENDENCY GRAPH:
  check_prerequisites
        ↓
  audit_generated_project  
        ↓
  verify_project  
        ↓ (optional --with-compose-up)
  wait_for_health + business_flow_check
        ↓
  Project internal scripts (anytime after generation)

5. CRITICAL GAPS BETWEEN SPECS AND IMPLEMENTATION
================================================================================

10 IDENTIFIED GAPS (Priority Order):

HIGH PRIORITY (Production Safety):

Gap 1: OpenAPI Export is Hardcoded Stub
  ├─ Spec: "FastAPI must provide scripts/export_openapi.sh to export to docs/openapi.json"
  ├─ Reality: Current script generates hardcoded JSON stub with endpoint list
  ├─ Risk: Exported schema doesn't match actual API signatures
  ├─ Fix: Extract from running /openapi.json endpoint or use Python introspection
  └─ Impact: API consumers get stale/incorrect documentation

Gap 2: Business Flow Validation is Minimal
  ├─ Spec: "Must be self-contained, repeatable, cover key role differences"
  ├─ Reality: Script exists and covers happy path but doesn't validate response content
  ├─ Risk: Wrong data shapes in responses go undetected
  ├─ Fix: Add field validation, status code verification
  └─ Impact: API contract violations not caught

Gap 3: Frontend Tests Not Checked for Fake Tests
  ├─ Spec: "Frontend tests must provide meaningful coverage"
  ├─ Reality: Backend has B.2 (all tests import app), frontend has nothing
  ├─ Risk: Empty tests or tests that don't import components pass
  ├─ Fix: Add check: "test files must import actual component/module"
  └─ Impact: Fake test files not detected

Gap 4: Request ID Middleware Only String-Checked
  ├─ Spec: "Request ID middleware must exist and add to logs"
  ├─ Reality: Only checks for string presence (could be commented code)
  ├─ Risk: Code shows request_id but it's not actually used
  ├─ Fix: Runtime verification that middleware adds ID to logs
  └─ Impact: Request tracing doesn't work in production

Gap 5: Metrics Route Templates Validation is Vague
  ├─ Spec: "Metrics path labels must use route templates, not actual URLs"
  ├─ Reality: Grep patterns are loose (scope.*route matches comments)
  ├─ Risk: High-cardinality metrics not caught
  ├─ Fix: Parse actual /metrics output, validate cardinality
  └─ Impact: Metrics explosion can crash Prometheus

MEDIUM PRIORITY (Quality):

Gap 6: Database/Redis Probes Only String-Checked
  ├─ Spec: "Health check must verify DB and Redis readiness"
  ├─ Reality: Only checks for SELECT 1 / .ping() strings
  ├─ Risk: Probes don't actually work
  ├─ Fix: Test that /health/ready fails when DB/Redis are down
  └─ Impact: Health checks are decorative, not functional

Gap 7: Frontend List Controls Binding Check is Incomplete
  ├─ Spec: "Input/select must have state binding connected to API"
  ├─ Reality: Only checks src/pages/, doesn't verify state→API call
  ├─ Risk: Form controls bound to state but not connected to API
  ├─ Fix: Extend to src/features/, verify state passed to API call
  └─ Impact: Decorative form controls not caught

Gap 8: Admin Bootstrap Only String-Checked
  ├─ Spec: "Admin must be initialized via seed script with env-var control"
  ├─ Reality: Only checks for bootstrap/seed_admin string
  ├─ Risk: Code mentions bootstrap but it doesn't work or isn't idempotent
  ├─ Fix: Add to business flow: verify admin login with env-var creds
  └─ Impact: Admin initialization fails in production

LOWER PRIORITY (Documentation):

Gap 9: Token Storage Risk Documentation Only Presence-Checked
  ├─ Spec: "localStorage token usage requires explicit risk doc"
  ├─ Reality: Accepts any mention of "localStorage"
  ├─ Risk: Doc says "localStorage" but no risk analysis or mitigation
  ├─ Fix: Verify both risk + mitigation strategy documented
  └─ Impact: Security risks not properly communicated

Gap 10: Dockerfile Non-Root User Only Presence-Checked
  ├─ Spec: "Backend must run as non-root user"
  ├─ Reality: Only checks for USER directive (could be commented)
  ├─ Risk: Directive exists but commented, container runs as root
  ├─ Fix: Verify actual runtime user via docker inspect
  └─ Impact: Container security misconfiguration not caught
