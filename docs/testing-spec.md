# Testing Spec

本文件是模板仓库的测试与验证总入口规范。目标不是要求每个项目都追求高覆盖率数字，而是要求 AI 生成的工程具备稳定、可重复、能暴露主链路风险的验证体系。

如果你只读一份测试规范，优先读本文件；如果需要补充上下文，再继续看以下文档：

- `docs/backend-spec.md`
- `docs/frontend-ui-spec.md`
- `docs/deployment-spec.md`
- `docs/generation-quality.md`
- `docs/business-checklist-template.md`

## 1. 目标

测试与验证必须同时满足三件事：

- 可执行：本地和 CI 都能稳定运行
- 可定位：失败时能快速指出是接口、状态流转、配置还是交互问题
- 可回归：修复后能重复验证，不依赖人工记忆

## 2. 通用原则

- 先验证关键业务闭环，再扩展次优先级场景
- 先保证真实状态流转，再增加实现细节断言
- 先写稳定可维护的测试，再追求表面覆盖率
- 先保证项目级验证命令成立，再谈附加测试框架

## 3. 默认质量门禁

每个生成项目至少必须通过以下检查：

- `docker compose config`
- `backend pytest`
- `backend ruff check`
- `frontend npm run build`
- `frontend npm run lint`

如果项目存在 `generated/<project-slug>/scripts/check_business_flow.sh`，则它也属于正式验证的一部分，而不是可选附录。

## 4. 测试分层

推荐至少按以下层次组织：

- 静态检查：`ruff check`、前端 `lint`
- 构建验证：前端 `build`、Compose 配置解析
- 接口测试：关键 API 的成功、失败、权限和异常路径
- 业务流测试：跨接口或跨状态流转的闭环验证
- 项目级脚本验证：容器启动后的关键业务动作检查

不要只做其中一层。例如只有构建通过而没有业务流测试，不算满足模板目标。

对独立工程还应同步验证项目级 AI 协作资产是否存在，至少包括 `AGENTS.md`、`CLAUDE.md`、`docs/ai-workflow.md` 与 review/fix 记录模板。

## 5. 后端测试硬规则

- 使用 `pytest`
- 测试位于 `generated/<project-slug>/backend/tests/`
- 至少覆盖关键接口、关键业务规则、关键状态流转和关键异常场景
- 必须覆盖未登录、无权限、非法输入、资源不存在、重复提交或冲突中的合理子集
- 测试应尽量围绕用户动作和业务规则组织，而不是只围绕函数存在与否

推荐覆盖重点：

- 创建 / 编辑 / 提交 / 审核 / 发布等核心写操作
- 列表、详情和状态查询的一致性
- 唯一约束、评分规则、归属权校验等核心业务规则
- 关键生命周期流转是否符合需求定义
- Redis 不可用、数据库异常、超时或重复提交等至少一类异常边界

## 6. 前端验证硬规则

- 前端必须保证 `npm run build`、`npm run lint` 与 `npm test -- --run` 或等价测试命令可通过
- 必须对照 `docs/frontend-ui-spec.md` 中的验收清单做一轮视觉与可用性自查
- 重点检查按钮换行、控件变形、长文本溢出、移动端筛选区布局、空态与错误态反馈
- 默认应引入前端测试框架，补关键页面、表单、空态/错误态或未登录引导测试；构建与 UI 自查不能替代测试
- 前端还应自查 ErrorBoundary、懒加载、统一 HTTP 错误处理与未登录/无权限交互提示

## 7. 项目级业务验证规则

- 每个生成项目都应输出 `generated/<project-slug>/docs/key-business-actions-checklist.md`
- 清单中的动作必须来自当前需求，而不是套用固定示例
- 建议提炼 3-5 个最关键业务动作
- 修复问题后，必须重新核对受影响动作的验证状态

如需求存在清晰主链路，应补：

- `generated/<project-slug>/scripts/check_business_flow.sh`

它至少应覆盖以下之一：

- 关键实体创建到可见
- 关键审批或状态流转
- 关键安装、发布、评价、分配或确认动作
- 关键角色之间的可见性与权限差异

## 8. 测试数据与环境隔离

- 测试必须尽量隔离环境，不依赖开发者本机手工准备数据
- 测试数据库、缓存命名和环境变量应与开发环境区分
- 不要让测试默认写入生产式配置或共享外部资源
- 固定种子数据、fixture 和工厂方法应服务于业务闭环，不要制造难维护的大量样板

## 9. CI / 自动化思路

模板默认不强制绑定特定 CI 平台，但生成工程应具备接入 CI 的基本条件：

- 命令无交互依赖
- 失败时退出码明确
- 环境变量清单完整
- 脚本顺序固定
- 构建、lint、pytest、compose 校验可串行执行
- 最佳情况下同步输出 CI 工作流，使这些检查能被平台直接复用

## 10. 常见错误

- 只写 happy path，没有失败或越权场景
- 只有接口单测，没有业务闭环验证
- 构建通过，但关键业务动作无法执行
- 修复代码后只重跑构建，不回归受影响主链路
- 测试依赖人工先启动一堆隐式服务或手工导入数据

## 11. 生成时的最小执行顺序

生成测试与验证时，建议至少按这个顺序执行：

1. 读取本文件 `docs/testing-spec.md`
2. 读取 `requirements/requirement.md`
3. 读取当前项目 OpenSpec 与 `docs/key-business-actions-checklist.md`
4. 读取 `docs/backend-spec.md`
5. 读取 `docs/frontend-ui-spec.md`
6. 读取 `docs/deployment-spec.md`
7. 先补业务动作清单，再补测试代码与验证脚本
8. 最后运行并记录 `compose`、`pytest`、`ruff`、`build`、`lint`

## 12. 验收清单

生成完成后，至少自查以下问题：

- 是否已经定义 3-5 个关键业务动作并写入项目级回归清单
- 后端测试是否覆盖成功、失败、权限和关键状态流转中的合理子集
- 是否至少覆盖一条数据库/Redis/超时等异常路径
- 后端测试数量不应少于 8 个关键用例；如果少于 8 个，必须在 review-log 中说明原因并补充业务流脚本覆盖
- 必须覆盖认证失败、越权访问、非法输入、重复/冲突、非法状态流转、依赖异常/限流触发中的至少 4 类
- 前端应默认接入 Vitest/Testing Library 或等价方案，至少提供页面 smoke、表单校验、错误态/空态、未登录引导中的合理子集
- CI 中应执行覆盖率命令或至少输出覆盖率报告；生产级目标后端核心业务覆盖率不低于 70%
- 前端是否完成构建、lint 与 UI 自查
- 项目级验证命令是否写入 `generated/<project-slug>/README.md`
- 如存在业务流脚本，是否已纳入正式验证路径
- 修复问题后，是否重新验证了受影响的主链路动作
