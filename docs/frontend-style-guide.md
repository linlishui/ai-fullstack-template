# Frontend Style Guide

本文件用于约束 AI 生成前端页面时的视觉与交互质量，避免只生成“可构建但缺乏层次”的默认样式。

## 1. 设计目标

- 页面必须先服务于业务闭环，再追求装饰
- 界面应有明确的信息层级、操作层级和状态反馈
- 风格要稳定统一，不允许每个页面各自为政
- 优先输出可以继续扩展的设计系统，而不是一次性页面拼装

## 2. 主题基础

生成前端时，至少先定义以下主题 token 或 CSS 变量：

- 颜色：主色、强调色、成功、警告、危险、中性色、边框色、背景层级
- 字体：标题、正文、辅助文本的字号与字重
- 间距：至少定义 4-6 个常用 spacing 档位
- 圆角：按钮、输入框、卡片、弹窗应有统一圆角规则
- 阴影：悬浮层、卡片、下拉菜单应有统一阴影层级
- 断点：移动端、平板、桌面端至少三档

禁止直接把所有颜色和 spacing 写死在单个组件中。

## 3. 页面层级

每个关键页面至少应体现：

- 页面标题与说明区
- 主操作区
- 内容主体区
- 辅助信息或筛选区
- 空态、错误态、加载态区域

不要把所有信息压成一块白底容器，也不要全站都是同一层级的卡片堆叠。

## 4. 组件约束

至少统一以下组件的视觉规则：

- Button：主按钮、次按钮、危险按钮、文本按钮
- Input / Select / Textarea：聚焦、错误、禁用样式
- Table / List：表头、行 hover、空态、分页或加载占位
- Card：标题、内容、操作区层级
- Modal / Drawer：标题、关闭、确认、取消层级
- Badge / Status：不同状态的语义颜色与可读性
- Alert / Toast：成功、失败、警告提示

## 5. 状态设计

每个关键页面都必须显式处理：

- 首次加载
- 局部加载
- 空数据
- 请求失败
- 表单校验失败
- 提交中
- 提交成功
- 禁用或无权限

不允许用户点击后没有反馈，不允许接口失败时静默。

## 6. 响应式要求

- 移动端下主要操作必须可达
- 表格类内容需要说明折叠、横向滚动或卡片化策略
- 多列布局在窄屏下必须重排
- 表单在移动端不能出现难以点击或被遮挡的问题

## 7. 审美底线

- 不要只输出默认白底、灰边、普通蓝按钮的无识别界面
- 不要所有页面共用完全一样的头图、卡片和布局节奏
- 不要把“精美”理解为堆渐变、堆阴影或堆动画
- 允许适度品牌化，但不能牺牲信息清晰度和可用性

## 8. 输出要求

生成项目时，建议在 `generated/<project-slug>/docs/frontend-ui-checklist.md` 中至少记录：

- 主题方向
- 页面与业务模块映射
- 关键组件复用策略
- 关键状态设计
- 移动端适配说明
- 仍待人工确认的体验风险
